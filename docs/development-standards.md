# 开发规范

## 基本原则

- 先读现有代码，再改动。
- 小步提交，保持每次变更可解释。
- 新功能优先落在明确模块内。
- 共享逻辑必须有清楚命名和边界。
- 文档和实现不一致时，应在同一次变更中修正。

## 语言和命名

- 代码命名优先使用英文。
- 面向用户的文案优先使用中文，除非产品风格需要混用。
- 文件名使用小写短横线，例如 `user-profile.ts`。
- 类型、类、组件使用 PascalCase，例如 `GoblinProfile`。
- 函数、变量使用 camelCase，例如 `loadLairMap`。
- 常量使用 UPPER_SNAKE_CASE，例如 `MAX_TUNNEL_DEPTH`。
- Go 包名使用小写短词，不使用下划线或短横线。
- Go 导出类型、函数和方法使用 PascalCase，非导出成员使用 camelCase。
- 数据库表名使用 snake_case 复数形式，例如 `goblin_profiles`。

## 目录和模块

- 不把所有代码放进 `utils`。
- 不把业务规则写在 UI 组件内部。
- 不跨模块引用内部文件；模块只暴露明确出口。
- Nuxt 业务页面放在 `apps/web`。
- OAuth 服务放在 `services/oauth`。
- 哥布林主业务服务放在 `services/goblin`。
- 公共前端组件放在 `packages/ui`。
- 跨端共享类型、常量和协议说明放在 `packages/shared`。

## 技术栈规范

- Nuxt 只负责页面渲染、用户交互和 API 调用。
- Gin 负责 OAuth 服务和哥布林主业务服务各自的 HTTP 路由、中间件和请求响应边界。
- Gorm 负责数据库访问，但复杂业务规则不得塞进 model hook。
- PostgreSQL 是主数据源。
- Redis 只存放缓存、限流、临时状态等可重建数据。
- Kafka 用于异步事件，不用于替代同步事务。
- Nginx 负责入口转发、静态资源策略和基础网关能力。
- Docker 是本地开发环境的标准入口。
- OAuth 账号、登录、Token 和第三方授权逻辑只能放在 OAuth 服务。
- 哥布林资料、内容、互动和业务权限只能放在哥布林主业务服务。
- 两个服务之间通过 API、Token 校验或 Kafka 事件协作，禁止跨库直接写入对方数据。

## 开发环境规范

- 项目默认在 Dev Container 中开发。
- 不要求开发者在宿主机手动安装 PostgreSQL、Redis、Kafka 或 Nginx。
- Node、Go、数据库、缓存、消息队列和网关依赖应由容器环境提供。
- 新增开发命令时，必须保证能在开发容器内执行。
- `.env.example` 必须覆盖本地容器环境所需的关键变量。
- 宿主机只应需要 Docker、支持 Dev Container 的编辑器或等价容器入口。

## 提交规范

推荐使用 Conventional Commits：

- `feat:` 新功能。
- `fix:` 修复问题。
- `docs:` 文档。
- `refactor:` 重构，不改变行为。
- `test:` 测试。
- `chore:` 工程杂项。

示例：

```text
docs: add project charter and architecture principles
feat: add lair map entry page
fix: prevent duplicate tunnel navigation
```

## 测试规范

测试覆盖按风险决定：

- 纯展示组件：可用轻量渲染测试或视觉检查。
- 业务规则：必须有单元测试。
- 权限、登录、支付、内容发布：必须有集成或端到端测试。
- 公共工具函数：应覆盖边界输入。
- Go service/usecase 层必须优先写单元测试。
- Repository 层涉及 SQL、迁移或事务时，应补集成测试。
- Nuxt 关键页面至少保留可手动复现的验证步骤，稳定后补端到端测试。

每次修改至少说明验证方式。如果暂时无法测试，应记录原因。

## 错误处理

- 用户可恢复的问题要给出可读提示。
- 开发者需要排查的问题要有日志和上下文。
- 不把底层错误原样暴露给用户。
- 权限失败、参数错误、服务异常要区分处理。

## 安全和治理

- 默认不信任客户端输入。
- 服务端必须重新校验权限。
- 用户生成内容需要预留审核、举报、屏蔽和限流能力。
- 敏感操作需要审计日志。
- 密钥、Token、Cookie、支付信息不得写入日志。

## UI 规范

- 视觉可以阴暗、粗粝、压迫，但交互必须清楚。
- 首页第一屏应是像素风 RPG 基地，让用户立即理解“哥布林巢穴”的主题。
- 首页应保持空旷基地感，通过场景地点承载模块入口，不用信息卡片堆满首屏。
- 场景中的可点击地点必须有 hover、focus、active 和移动端触摸反馈。
- 每个入口都应有可读标签、tooltip 或等价提示，不能只靠装饰图猜功能。
- 导航可以设计成洞穴、岔路、地图等隐喻，但不能让核心功能不可达。
- 移动端必须避免遮挡、溢出和不可点击区域。
- 重要操作必须有明确反馈。
