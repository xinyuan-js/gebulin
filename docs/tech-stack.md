# 技术栈

## 总览

哥布林巢穴采用前后端分离、容器化、本地可复现的基础架构。

- 前端：Vue + Nuxt。
- 后端：Go + Gin + Gorm，拆分为 OAuth 服务和哥布林主业务服务。
- 数据库：PostgreSQL。
- 缓存和会话辅助：Redis。
- 事件和异步消息：Kafka。
- 网关和反向代理：Nginx。
- 本地开发和部署基础：Docker。

## 前端

前端使用 Nuxt 作为主 Web 应用框架。

职责：

- 页面路由。
- SSR 或混合渲染能力。
- 用户界面和交互。
- 调用后端 API。
- 处理前端鉴权状态和基础缓存。

约定：

- 主应用放在 `apps/web`。
- Vue 组件使用单文件组件。
- 页面级路由遵守 Nuxt 约定。
- 跨页面 UI 组件沉淀到 `packages/ui`。
- 与后端共享的类型和常量优先放在 `packages/shared`。

## 后端

后端使用 Go 作为主语言，Gin 作为 HTTP 框架，Gorm 作为 ORM。后端从项目早期就拆成两个服务：

- `services/oauth`：独立 OAuth 和身份认证服务。
- `services/goblin`：哥布林巢穴主业务服务。

职责：

- OAuth 服务负责账号、登录、授权、Token、身份安全。
- 哥布林主业务服务负责资料、内容、互动、巢穴分区和业务权限。
- 两个服务都通过 REST API 暴露能力。
- 两个服务都可以按需使用 PostgreSQL、Redis 和 Kafka，但数据所有权必须隔离。

约定：

- OAuth 服务放在 `services/oauth`。
- 哥布林主业务服务放在 `services/goblin`。
- HTTP 层只做请求解析、响应封装和中间件处理。
- 业务逻辑放在 service/usecase 层。
- 数据库访问放在 repository 层。
- Gorm model 不直接替代领域模型；复杂业务应保持独立领域结构。
- OAuth 服务不得包含哥布林主业务逻辑。
- 哥布林主业务服务不得签发或管理 OAuth Token。

## PostgreSQL

PostgreSQL 是主持久化数据库。

使用范围：

- 用户。
- 内容。
- 评论。
- 权限。
- 审计日志。
- 需要事务一致性的业务数据。

约定：

- 所有结构变更必须通过迁移文件管理。
- 不在应用启动时静默修改生产表结构。
- 重要字段必须考虑索引、唯一约束和外键关系。

## Redis

Redis 用于高频、短生命周期或可重建数据。

使用范围：

- 会话辅助信息。
- 限流计数。
- 缓存。
- 短期验证码或临时令牌。
- 分布式锁，只有确有必要时使用。

约定：

- Redis 数据必须设置清晰 key 前缀。
- 临时数据默认设置 TTL。
- 不把 Redis 当作唯一持久化来源。

## Kafka

Kafka 用于异步事件和跨模块解耦。

使用范围：

- 用户行为事件。
- 内容发布事件。
- 审核事件。
- 通知事件。
- 未来排行榜、积分、任务系统的异步计算。

约定：

- Topic 命名需要体现领域和事件类型。
- 事件结构应版本化。
- 消费者必须考虑重复消费。
- 关键链路不能只依赖异步事件完成强一致操作。

## Nginx

Nginx 作为入口网关和反向代理。

职责：

- 转发 Web 和 API 请求。
- 静态资源缓存策略。
- 基础安全头。
- 请求体大小限制。
- 生产环境 TLS 终止，视部署方案确定。

## Docker

Docker 和 Dev Container 是本地开发环境和部署基础。项目默认不要求开发者在宿主机手动安装 PostgreSQL、Redis、Kafka、Nginx 等运行依赖。

约定：

- 提供 `.devcontainer` 开发容器配置。
- 提供 `docker-compose` 本地开发编排。
- Web、OAuth、哥布林主业务、PostgreSQL、Redis、Kafka、Nginx 应能在本地一键启动。
- 容器配置通过环境变量注入。
- 生产镜像应尽量小，避免包含开发依赖。
- 后续开发命令、测试命令和迁移命令默认在开发容器内执行。
