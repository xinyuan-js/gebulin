# 架构原则

## 总原则

哥布林巢穴应被设计成“多入口、多分区、多模块”的系统。每个功能模块都像一个洞穴分支：可以独立成长，也可以通过明确接口接入主巢穴。

基础技术栈为 Nuxt Web 应用、Go Gin 后端服务、PostgreSQL 主库、Redis 缓存、Kafka 消息队列、Nginx 网关和 Docker 容器化环境。后端拆分为独立 OAuth 服务和哥布林主业务服务，架构设计应围绕这两个边界做清晰分工，而不是把身份认证和主业务混在一个应用里。

## 分层

推荐按职责分层，而不是按页面随意堆放：

- 表现层：Nuxt 页面、Vue 组件、布局、交互。
- API 层：OAuth 服务和哥布林主业务服务各自的 Gin 路由、中间件、请求解析、响应封装。
- 领域层：业务概念、规则、权限、状态变化。
- 数据层：Gorm repository、PostgreSQL、Redis、Kafka producer/consumer。
- 平台层：配置、日志、鉴权、监控、Nginx、Docker、部署。

Nuxt 可以承载前端状态和展示逻辑，但核心权限和业务规则必须在 Go API 中重新校验。

## 服务边界

后端分为两个独立服务：

- OAuth 服务：负责账号、登录、授权、Token、第三方身份扩展和身份安全审计。
- 哥布林主业务服务：负责哥布林资料、内容、互动、巢穴分区、称号、成就和运营业务。

边界原则：

- OAuth 服务拥有账号和认证数据。
- 哥布林主业务服务拥有业务档案和内容数据。
- 两个服务通过 API、Token 校验和 Kafka 事件协作。
- 不允许一个服务直接写入另一个服务拥有的数据表。
- 早期可以共用 PostgreSQL、Redis、Kafka 实例，但必须通过 database/schema、key 前缀和 topic 命名隔离。

## 模块原则

每个功能模块应包含：

- 清晰入口：路由、导航项或 API 前缀。
- 独立模型：自己的类型、状态和业务规则。
- 显式依赖：依赖公共能力时通过共享库或接口接入。
- 可关闭能力：未来应允许通过配置关闭实验性模块。

模块之间禁止：

- 互相直接读取内部状态。
- 复制粘贴业务规则。
- 通过隐式全局变量通信。
- 在 UI 组件中绕过权限或数据边界。

## 首页和模块入口

哥布林主业务首页是像素风 RPG 基地，不是普通信息流。首页中的可点击地点应被当成模块入口管理。

入口模型至少应包含：

- 唯一标识。
- 显示名称。
- 路由。
- 开放状态。
- 权限要求。
- 场景坐标或布局区域。

早期入口可以由 Nuxt 前端静态配置；当入口状态、权限和活动运营复杂后，应迁移到哥布林主业务服务，由后端返回当前用户可见和可进入的入口列表。

## 领域概念

早期建议保留这些核心概念：

- `User`：用户。
- `Account`：OAuth 服务拥有的账号身份。
- `GoblinProfile`：项目风格下的用户身份包装。
- `Lair`：巢穴或分区。
- `Tunnel`：入口、路由或跨模块连接。
- `LairEntrance`：首页基地中的可点击模块入口。
- `Post`：内容主体。
- `Comment`：互动内容。
- `Badge`：称号、成就或身份标记。
- `InventoryItem`：未来可能出现的道具或收藏物。

这些名字可以在实现时按技术栈调整，但领域边界应保持稳定。

## 扩展策略

新增功能时，优先判断它属于哪一类：

- 新页面：只影响路由和表现层。
- 新模块：需要独立目录、模型、服务和测试。
- 新公共能力：应沉淀到共享库，例如鉴权、日志、错误处理。
- 新平台能力：应进入基础设施文档，例如队列、缓存、监控。

当功能需要异步处理时，优先通过 Kafka 发布领域事件；当功能需要短期状态或高频读取时，优先评估 Redis；当功能需要事务和长期保存时，必须落到 PostgreSQL。

## 配置策略

所有环境差异应通过配置表达：

- 本地开发配置。
- 测试配置。
- 生产配置。
- 功能开关。
- 第三方服务密钥。

密钥不得提交到仓库。仓库可以提供 `.env.example`，但真实 `.env` 应被忽略。
